For this project, I worked in a team with two other people, Max Miranda and Mudabbir Khan, and my role was mainly one of architecting the framework of the entire project and coordinating between different developers.

● I designed and built out the entire front-end, including the "Availability Selector" (UICollectionView) and the "Overall Calendar View" with multiple people's availability overlaid.
● I structured the database (Firebase), implemented multiple back-end functions, one of which allowed users to share and join events (Max Miranda wrote most of the functions which made the API calls), and put the entire project together.

All this, including ideation and wireframing, was done in three weeks and at the end of three weeks, we deployed MeetMe onto the Apple App Store and showcased our app at the Berkeley App Fair.

Project Architecture

Model Classes

MeetMe was built following the MVC (Model, View, Controller) paradigm, and for our core functionality, we had three model classes: User, Meetup, Calendar. The Calendar model contains the information of an individual calendar, and is linked to a User. A Meetup (which can have multiple users and hence multiple calendars), is a wrapper class with these calendars and other information such as overlapping availabilities.

Concept

Oh boy this is my favorite part: essentially, MeetMe retrieves the user's availability through Google Calendar's freebusy API: giving the API a set of Calendar IDs (representing a particular user's different calendars), the API returns to us the free and busy times of the user. MeetMe uses this information to populate the Calendar model, which are then overlaid one after the other in a Meetup model to show common free times.

Authentication is two-layered: the first is in-app authentication, when the user creates an account with us, and this is handled by the Firebase Auth Console; the second is authentication with GoogleSignIn, using OAuth 2.0, which allows us access to a user's Calendar IDs corresponding to his different Google Calendars. This information is stored on Firebase, and presented in the form of a UICollectionView.

Also worth mentioning is JTAppleCalendar, the CocoaPod that we used to implement the calendar from which a user can select available dates when creating an event.

Future Extensions

We are not done . In fact, we are far from done. Apart from Google Calendar, we are planning to expand functionality to include Apple Calendar. We plan to refactor by porting the core database functions onto a REST API, in preparation for future migration onto other platforms (Android, Desktop). We plan to do more with the Google Calendar API, and actually include information about what users are busy with (as opposed to just noting that they are busy). There's a lot more, I'm excited, and I hope, (reading this), that you are too :-)
