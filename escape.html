Implementation

Overview

Languages: Java
Frameworks: IntelliJ
―
I worked with one other person, Andrew Gao, for this project, but there was no clear division of labor because we both worked on multiple parts of the project together.

Notably, however, I designed and built the Random World Generating Engine that we eventually used, and the Load/Save feature and Andrew implemented the loading screen and wrote the Line-of-sight feature that was key to our game mechanic.

All other features, including the interactivity, health bars, mouse-hover descriptions, we both worked on together.

The Random World Generator, including initial ideation and structure, was completed in a week, while the interactivity and game mechanics were completed the week after.

Concept

Overview

We began the project with a collection of TETile objects, which was given to us as part of what was provided in CS 61B, Berkeley's data structures class. There were different tiles, and each tile represented a unit of space in the rendered world - we then figured out how to put these tiles together in different ways to create the game.

Random World Generation

Essentially, the world consisted of rooms and corridors, and each time a new seed was entered, random numbers of rooms/corridors of random sizes/lengths were generated.

To go into a bit more detail, we had a function which generated the random number of rooms/corridors, whose output fed into functions that created the rooms/corridors (WITHOUT the walls). These room/corridor-creating function created these structures with different sizes/lengths, and once everything was made, we had a final wrapper function that created walls around everything.

Interactivity

Interactivity was achieved through a Listener that listened for particular keystrokes. Certain keystrokes (WASD, for instance) corresponded to certain actions, and when these keystrokes were pressed, the world was re-rendered accordingly.

Save/Load

One of the most challenging parts of the project, save/load was achieved by having our world object extend serializable. This allowed the world to be written to a .txt file, which we can later read from to re-render the world.

Our initial, naive approach was to iterate over every single tile in the entire world, and save a digit corresponding to each tile in a .txt file: so a black tile might be '0', a flower might be '1', a player might be '2' etc.

However, not only was this not as elegant, but saving in this manner also did not preserve the randomness native to the original file.

Takeaways

As one of the largest software projects I've built with a partner, this project has taught me so much about managing complexity and handling scale in a project.

● It taught me and reinforced the idea of layers of abstraction, of how it's helpful to create the basic building blocks, first, before layering functionality on top of these building blocks.
● It taught me the importance of time management, especially since this project happened during a particularly busy period of my school year.
● It taught me a lot about communication and working with another engineer, something that I don't often chance to do in a classroom environment.
